# Dataset Statement - TunnelSight GAMIS

This document defines the dataset strategy for TunnelSight AI, including available project assets, controlled field data, synthetic data and validation datasets.

## D1. Dataset Register

| Dataset name | Source | Data type | Volume | Labels | Rights/status | Processing | Limitations |
|---|---|---|---:|---|---|---|---|
| TunnelSight GLB model set | Local project assets | GLB meshes | 3 model files | Inspection markers added in application layer | Internal project/demo use; provenance to be verified before public dataset release | Imported into PlayCanvas viewer | Used for interface, navigation and marker workflow validation |
| Seeded risk-marker set | Created by development team | Marker/annotation JSON in app code | 3 seeded markers | High, Medium, Warning | Team-created annotation data | Rendered as 3D markers and labels | Used to exercise the inspection and reporting workflow |
| User-created inspection markers | Created in viewer | Manual marker annotations | Browser session data | Warning default; editable in future workflow | User-generated; consent required for real site data | Created in browser | Persistence handled through backend/API workstream |
| Controlled mine/tunnel capture dataset | Field capture with site permission | Images/video/point cloud/mesh | Pilot acquisition set | Cracks, water ingress, loose rock, unsupported roof, safe zone | Site permission and data owner approval required | Frame extraction, cleaning, annotation | Requires controlled capture conditions and expert labels |
| Synthetic tunnel hazard dataset | Generated by project team or licensed tools | Rendered images/meshes | Iterative synthetic set | Simulated cracks, seepage, rockfall zones | Team-generated or generated with licensed tools | Synthetic rendering and labels | Supports early model development and stress testing |
| Public third-party mining/geology data | Evaluated public sources | Images/point clouds/metadata | Dataset-dependent | Depends on dataset | Licence must permit research/development use | Normalisation and relabelling | Used only where licence and relevance are verified |

## D2. Real Mine Data

Real mine data is handled as controlled-access operational information. Mine images, scans, coordinates and production information require permission from the owner/operator and are protected as confidential project data.

## D3. Synthetic and Simulated Data

Synthetic and simulated records are used for interface validation, early model development, hazard examples and stress testing. These records are separated from field-captured data so that validation metrics can distinguish real, simulated and public-source evidence.

## D4. Data Still Required

For AI validation, the project requires:

- Controlled images or videos of underground mine/tunnel surfaces.
- Expert-labelled examples of cracks, loose rock, water ingress, unsupported roof/crown, entrance/exit zones, and safe zones.
- Negative examples with no visible hazard.
- Lighting and capture-condition variation.
- Validation set separate from training data.

## D5. Annotation Plan

Initial annotation categories:

- Crack.
- Loose rock.
- Water ingress.
- Unsupported roof/crown.
- Wall deformation.
- Floor/invert damage.
- Entrance/exit.
- Safe zone.
- Unknown/needs expert review.

Annotation fields:

```json
{
  "label": "crack",
  "severity": "High|Medium|Warning",
  "confidence": 0.0,
  "source": "manual|ai",
  "review_status": "unreviewed|accepted|rejected",
  "reviewer_role": "mining_engineer|geologist|safety_officer"
}
```

## D6. Data Governance

The project will apply:

- Uploader consent confirmation.
- Ownership/permission declaration.
- No public release of confidential mine scans.
- Anonymisation/removal of unnecessary location details where possible.
- Encrypted storage in future backend.
- Data retention and deletion policy before pilot deployment.

## D7. Dataset Limitations

- Project model assets support the inspection workflow and interface evaluation.
- Field validation depends on legally obtained, expert-labelled tunnel or mine data.
- AI model performance is reported only after labelled validation data is prepared.
- False negatives are high-risk and must be measured directly during evaluation.
